## 1. 项目概述

### 1.1 项目背景

建设一个多租户企业管理系统平台，支持多家公司（租户）同时使用。各公司在平台内独立运营，可管理自身组织架构（含多级分公司）、自定义角色权限、管理人员账号与权限。平台方通过超级管理员后台统一管理所有入驻公司。

### 1.2 核心目标

| 目标 | 描述 | 衡量标准 |
|------|------|----------|
| 租户入驻 | 超级管理员可为新公司创建账号并初始化 | 创建公司后，公司管理员可立即登录 |
| 组织自管理 | 公司内部可管理分公司、部门、角色、权限、人员 | 无需超级管理员干预即可完成内部组织调整 |
| 数据完全隔离 | 各公司的数据、操作、界面、业务流程彼此不可见、不可访问 | 公司 A 的任何接口调用无法获取公司 B 的数据 |
| 可扩展性 | 预留标准接口，后续可接入 OA、业务系统等模块 | 第三方模块按接口规范对接后无需修改核心代码 |

### 1.3 用户角色体系概览

```
平台级：
  └── 超级管理员（平台方）  →  管理所有公司

公司级：
  └── 公司管理员（公司方）  →  管理本公司所有资源
        └── 自定义角色 1    →  由公司管理员创建，灵活配置权限
        └── 自定义角色 2
        └── 自定义角色 N
              └── 普通员工    →  使用被分配的功能
```

### 1.4 用例总览图

```
超级管理员
  ├── 创建/停用/删除公司
  ├── 管理公司套餐/有效期
  ├── 查看所有公司概览
  └── 平台级审计日志

公司管理员
  ├── 创建/管理分公司
  ├── 创建/管理部门
  ├── 自定义角色并赋权
  ├── 管理公司人员（增删改查、赋角色）
  ├── 配置公司基础信息
  └── 查看公司级操作日志

普通员工（由公司自定义角色决定）
  ├── 登录并使用被授权的功能模块
  └── 在权限范围内操作业务数据
```

## 2. 系统架构概览

### 2.1 多租户架构模式

采用 **共享数据库 + 租户 ID 隔离** 模式（即一个数据库实例，所有表均有 `tenant_id` 字段区分租户）。

```
┌─────────────────────────────────────────────────────────────┐
│                        前端层                               │
│   ┌──────────────┐  ┌──────────────┐  ┌──────────────┐     │
│   │ 超级管理后台  │  │ 公司管理后台  │  │   员工端     │     │
│   └──────┬───────┘  └──────┬───────┘  └──────┬───────┘     │
└──────────┼──────────────────┼──────────────────┼───────────┘
           │                  │                  │
┌──────────┴──────────────────┴──────────────────┴───────────┐
│                      API 网关层                             │
│   ┌──────────────────────────────────────────────────────┐ │
│   │   统一认证  │  租户识别  │  权限校验  │  限流/日志   │ │
│   └──────────────────────────────────────────────────────┘ │
└──────────────────────────┬─────────────────────────────────┘
                           │
┌──────────────────────────┴─────────────────────────────────┐
│                      业务服务层                             │
│   ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌───────────┐ │
│   │ 租户管理 │  │ 组织架构 │  │ 角色权限 │  │ 人员管理  │ │
│   └──────────┘  └──────────┘  └──────────┘  └───────────┘ │
│   ┌──────────┐  ┌──────────┐                               │
│   │ 操作日志 │  │ 扩展服务 │（预留：OA/业务/财务等）       │
│   └──────────┘  └──────────┘                               │
└──────────────────────────┬─────────────────────────────────┘
                           │
┌──────────────────────────┴─────────────────────────────────┐
│                       数据层                                │
│   ┌──────────────────────────────────────────────────────┐ │
│   │    共享数据库（所有表含 tenant_id）                    │ │
│   └──────────────────────────────────────────────────────┘ │
│   ┌──────────┐  ┌──────────┐  ┌──────────────────────────┐ │
│   │  Redis   │  │   OSS    │  │    文件存储（按租户分桶）  │ │
│   │ 缓存隔离 │  │ 对象存储 │  │                          │ │
│   └──────────┘  └──────────┘  └──────────────────────────┘ │
└────────────────────────────────────────────────────────────┘
```

### 2.2 租户识别机制

- 登录时根据账号归属确定 `tenant_id`
- 所有 API 请求在网关层注入当前用户上下文（`tenant_id`、`user_id`、`role_codes`）
- 数据访问层（DAO / ORM）自动附加 `WHERE tenant_id = ?` 条件，防止跨租户数据泄露
- 超级管理员的 `tenant_id` 为空或为特殊标记（`tenant_id = 0`），可跨租户访问

### 2.3 技术栈建议

| 层级 | 技术选项 | 说明 |
|------|----------|------|
| 后端框架 | Spring Boot / Django / Go Gin | 根据团队技术栈选择 |
| 数据库 | MySQL 8.0+ / PostgreSQL 15+ | 支持 JSON 字段，便于扩展属性 |
| 缓存 | Redis 7+ | 租户隔离通过 key 前缀：`{tenant_id}:{业务key}` |
| 对象存储 | MinIO / 阿里云 OSS / 腾讯云 COS | 按 `{tenant_id}/` 路径隔离 |
| 前端 | Vue 3 + Element Plus / React + Ant Design | 已用 ardot 初步搭建界面，可在此基础上开发 |
| API 鉴权 | JWT（含 tenant_id, user_id, role_codes） | Token 自动过期 + 刷新机制 |

## 3. 功能模块详细规格

---

### 3.1 超级管理后台

**访问入口**：独立后台，仅超级管理员可登录（与公司端入口分离或通过角色区分）。

#### 模块 3.1.1：公司（租户）管理

##### 3.1.1.1 公司列表

**功能描述**：展示所有已入驻的公司列表，支持搜索、筛选、分页。

**列表字段**：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| 公司编号 | 自动生成 | 系统唯一标识，如 `C00001` |
| 公司名称 | 文本 | 公司全称 |
| 公司简称 | 文本 | 用于界面显示 |
| 状态 | 枚举 | `启用` / `停用` / `待激活` |
| 管理员账号 | 文本 | 该公司管理员登录账号 |
| 管理员手机号 | 手机号 | 用于联系和短信通知 |
| 入驻时间 | 日期时间 | 账号创建时间 |
| 有效期截止 | 日期 | 到期后自动停用 |
| 分公司数量 | 数字 | 该公司下分公司总数 |
| 人员数量 | 数字 | 该公司下所有人员总数 |
| 操作 | 按钮组 | 查看/编辑/停用/续期/删除 |

**搜索筛选条件**：
- 公司名称（模糊搜索）
- 公司编号（精确搜索）
- 状态（下拉选择：全部/启用/停用/待激活）
- 入驻时间范围（日期区间选择器）

**排序**：默认按入驻时间倒序，支持按公司编号、分公司数量排序。

##### 3.1.1.2 创建公司

**功能描述**：超级管理员为一家新公司创建租户账号。

**录入项**：

| 字段名 | 必填 | 校验规则 | 说明 |
|--------|------|----------|------|
| 公司全称 | 是 | 2-50 字符，不可与已有名称重复 | 营业执照上的全称 |
| 公司简称 | 是 | 2-20 字符 | 登录页、界面显示用 |
| 统一社会信用代码 | 否 | 18 位字符 | 可选，后续对接工商接口 |
| 管理员登录账号 | 是 | 6-20 位字母数字，不可重复 | 公司管理员的登录用户名 |
| 管理员登录密码 | 是 | 8-20 位，需含大小写字母+数字 | 初始密码，首次登录强制修改 |
| 管理员姓名 | 是 | 2-10 字符 | 管理员真实姓名 |
| 管理员手机号 | 是 | 11 位手机号格式 | 用于找回密码、通知 |
| 管理员邮箱 | 否 | 邮箱格式 | 用于通知 |
| 有效期 | 是 | 日期选择器 | 公司账号的截止使用日期 |
| 最大人员数 | 否 | 正整数 | 限制该公司可创建人员上限，留空不限制 |
| 最大分公司数 | 否 | 正整数 | 限制该公司可创建分公司上限，留空不限制 |
| 备注 | 否 | 200 字符以内 | 内部备注信息 |

**创建流程**：
1. 填写表单 → 提交
2. 系统校验所有字段
3. 生成公司编号（规则：`C` + 年月日 + 3 位流水号，如 `C20260625001`）
4. 创建 `company` 记录，`status = 待激活`
5. 自动创建该公司默认超级管理员角色（公司内最高权限）
6. 创建管理员用户记录，关联该公司，角色为公司超级管理员
7. 初始化该公司的基础数据（如默认部门）
8. 返回创建成功，可选择发送短信/邮件通知管理员

##### 3.1.1.3 编辑公司信息

**可编辑项**：
- 公司全称、公司简称
- 管理员姓名、手机号、邮箱
- 有效期
- 最大人员数、最大分公司数
- 状态（启用/停用）
- 备注

**不可编辑项**：
- 公司编号（系统生成，不可改）
- 管理员登录账号（创建后不可改）
- 统一社会信用代码（如需修改，走特殊流程）

##### 3.1.1.4 停用/启用公司

**停用**：
- 操作前弹出二次确认："停用后该公司所有人员将无法登录，确定继续？"
- 停用后：该公司所有用户 Token 立即失效，无法登录
- 停用后：该公司数据保留不删除
- 记录操作日志

**启用**：
- 恢复公司的正常使用状态
- 该公司用户可重新登录
- 记录操作日志

##### 3.1.1.5 删除公司

**删除规则**：
- 弹出二次确认，要求输入公司全称确认
- 执行**逻辑删除**（标记 `is_deleted = true`），不使用物理删除
- 删除后该公司所有用户无法登录
- 删除后数据保留（用于审计追溯）
- 记录操作日志

**物理删除**（后续版本考虑）：超级管理员手动发起 + 二次验证 + 冷却期后才真正清除数据。

##### 3.1.1.6 公司详情

点击公司行进入详情页，展示：

**基础信息卡片**：
- 公司编号、全称、简称、统一社会信用代码
- 状态、入驻时间、有效期截止（逾期标红）
- 管理员信息（姓名、手机号、邮箱、最后登录时间）

**统计卡片**：
- 分公司总数
- 人员总数
- 角色总数
- 近 30 天活跃用户数

**操作历史 Tab**：
- 该公司的关键操作记录（谁、什么时候、做了什么）

#### 模块 3.1.2：平台概览仪表板

**功能描述**：超级管理员首页，展示平台整体运营数据。

**展示内容**：

| 指标 | 说明 |
|------|------|
| 入驻公司总数 | 统计所有 status=启用 的公司 |
| 本月新增公司 | 本月创建的租户数量 |
| 平台总用户数 | 所有公司人员总数 |
| 活跃公司数 | 近 7 天有用户登录的公司数 |
| 即将到期公司 | 有效期在未来 30 天内的公司，标红提醒 |

**图表**：
- 近 6 个月入驻公司趋势（折线图）
- 公司规模分布（柱状图：按人员数量分段统计）

### 3.2 公司管理后台（租户端）

**访问入口**：公司管理员和员工登录后的主界面。

#### 模块 3.2.1：公司基础信息管理

**功能描述**：公司管理员查看和编辑本公司基础信息。

**可查看项**：

| 字段 | 说明 |
|------|------|
| 部门名称 | 必填，同层级不可重名 |
| 部门编码 | 自动生成或手动输入，唯一 |
| 上级部门 | 树形选择，为空表示一级部门 |

**功能清单**：

| 功能 | 说明 |
|------|------|
| 查看部门树 | 左侧树形展示所有部门，可展开/折叠 |
| 新增部门 | 选择上级部门后新建子部门；或在根目录新建一级部门 |

| 删除部门 | 部门下无人员和子部门时方可删除；否则提示先迁移 |
| 移动部门 | 拖拽或选择目标父部门进行迁移 |
| 停用部门 | 停用后该部门下人员不做变更，但不能再将新人员归入 |
| 批量导入 | 支持 Excel 模板导入部门结构 |

**约束**：
- 部门层级最多 5 级
- 删除部门前检查：无子部门 + 无关联人员 + 无关联分公司
- 停用的部门在人员选择器中不显示

### 3.3 分公司管理

**核心概念**：分公司是公司下的独立组织单元。分公司具有独立的数据空间，但归属公司统一管理。

#### 模块 3.3.1：分公司架构

**分公司属性**：

| 字段 | 说明 |
|------|------|
| 分公司名称 | 必填，公司内唯一 |
| 分公司编码 | 自动生成，如 `B001` |
| 所属公司 | 自动关联当前公司 |
| 分公司类型 | 枚举：`独立运营` / `挂靠管理` |
| 所在地区 | 省/市/区（三级联动选择器） |
| 详细地址 | 文本 |
| 负责人 | 关联公司人员 |
| 联系电话 | 手机号格式 |
| 状态 | 启用 / 停用 |

**功能清单**：

| 功能 | 说明 |
|------|------|
| 分公司列表 | 表格展示，支持搜索（名称、编码）、筛选（状态、地区） |
| 新增分公司 | 公司管理员创建分公司，填写所属部门、负责人等 |
| 编辑分公司 | 修改分公司基础信息 |
| 停用/启用分公司 | 停用后该分公司数据只读 |
| 删除分公司 | 仅无关联业务数据时可删除 |

**分公司与部门的关系**：
- 一个分公司可以关联一个或多个部门
- 部门先创建（见 3.2.2），然后在分公司设置中选择所属部门
- 分公司的人员由其关联部门的人员组成

**分公司数据隔离**：
- 分公司之间数据默认隔离（业务模块中的数据需标记 `branch_id`）
- 公司管理员可查看所有分公司数据
- 分公司角色权限仅管辖所属分公司范围内的数据

#### 模块 3.3.2：分公司配额管理

**功能描述**：公司管理员可以为每个分公司设置资源上限。

| 配额项 | 说明 |
|--------|------|
| 最大人员数 | 该分公司可容纳的人员上限 |
| 最大存储空间 | 该分公司可用的文件存储上限（后续对接 OSS） |

### 3.4 角色与权限体系（RBAC）

#### 3.4.1 权限模型设计

采用 **RBAC（基于角色的访问控制）** 模型，三层结构：

```
权限点 (Permission)
    ↓ 多对多关联
角色 (Role)
    ↓ 多对多关联
用户 (User)
```

**权限点说明**：

| 组成部分 | 说明 | 示例 |
|----------|------|------|
| 权限模块 | 功能所属模块 | 分公司管理、人员管理、角色管理、系统设置 |
| 权限操作 | 允许的操作类型 | 查看、新增、编辑、删除、导出、审核 |
| 权限资源 | 作用范围 | 全部 / 仅本公司 / 仅本分公司 / 仅本人 |

**预置权限点清单（V1.0）**：

| 模块 | 权限点 | 编码 | 说明 |
|------|--------|------|------|

#### 3.4.2 角色创建与管理

##### 预置角色

系统为每个公司预置以下不可删除的角色：

| 角色名称 | 角色编码 | 说明 | 权限范围 |
|----------|----------|------|----------|
| 公司超级管理员 | `company_super_admin` | 公司最高权限角色，创建公司时自动生成 | 所有权限 + scope:all |
| 普通员工 | `employee_default` | 新员工默认角色（基础权限） | 个人中心相关 |

##### 公司自定义角色

**创建流程**：
1. 公司管理员进入"角色管理"页面
2. 点击"新建角色"
3. 填写角色信息：

| 字段 | 说明 |
|------|------|
| 角色名称 | 必填，公司内唯一，如"财务主管"、"项目经理" |
| 角色编码 | 自动生成（字母+数字），或手动输入，唯一 |
| 角色描述 | 可选，说明该角色的用途 |
| 是否启用 | 开关，关闭后该角色下的用户权限不生效 |
| 数据范围 | 下拉选择：全部 / 仅本分公司 / 仅本部门 / 仅本人 |

4. 配置权限：
   - 以权限树形式展示所有可用权限点（模块 → 权限点）
   - 勾选复选框授予权限
   - 支持"全选模块"快捷操作
5. 保存后角色立即生效（已持有该角色的用户刷新或下次登录后获得新权限）

**角色编辑**：
- 修改角色名称、描述、权限配置
- 权限变更后已登录用户需重新获取 Token 或刷新权限缓存
- 不可修改角色编码

**角色删除**：
- 仅可删除公司自定义角色（预置角色不可删除）
- 删除前检查：该角色下是否有关联用户
  - 有关联用户 → 提示"该角色关联了 N 个用户，请先移除关联或更换角色"
  - 无关联用户 → 二次确认后删除

**角色列表视图**：

| 字段 | 说明 |
|------|------|
| 角色名称 | 显示名称 |
| 角色编码 | 系统内部编码 |
| 关联用户数 | 持有该角色的用户数量 |
| 权限摘要 | 展示该角色拥有的顶级权限模块名称（3 个以内 +"等"） |
| 数据范围 | 全部 / 本分公司 / 本部门 / 仅本人 |
| 是否预置 | 标记为"系统预置"或"自定义" |
| 状态 | 启用 / 停用 |
| 创建时间 | 角色创建时间 |
| 操作 | 编辑 / 删除（仅自定义） |

- 停用后该用户无法登录，Token 立即失效

- 查看/编辑个人基本信息（姓名、手机号、邮箱、头像）

- 查看我的角色和权限列表（只读）
- 查看我的登录历史（最近 20 条登录记录）

### 3.6 数据隔离机制

这是系统的核心要求之一，需从以下维度确保隔离：

#### 3.6.1 数据库层隔离

**强制规则**：所有业务表必须包含 `tenant_id` 字段。

```sql
-- 以部门表为例
CREATE TABLE dept (
    id          BIGINT PRIMARY KEY,
    tenant_id   BIGINT NOT NULL,        -- 租户ID，所有查询必须带上
    company_id  BIGINT NOT NULL,        -- 等同于 tenant_id，便于理解
    name        VARCHAR(50) NOT NULL,
    parent_id   BIGINT DEFAULT 0,
    -- ...
    INDEX idx_tenant (tenant_id)        -- 必须建索引
);
```

**DAO 层约束**（开发清单）：
- [ ] 所有查询方法必须显式添加 `tenant_id` 条件
- [ ] 所有写入方法必须显式赋值 `tenant_id`
- [ ] 统一封装基础 DAO 方法，自动注入 `tenant_id`
- [ ] 禁止在业务代码中使用原生 SQL 直连数据库（绕过 ORM）

#### 3.6.2 缓存层隔离

Redis Key 命名规范：

```
{tenant_id}:{模块}:{业务标识}
```

示例：
```
10001:user:token:abc123       -- 公司 10001 的用户 Token
10001:branch:list             -- 公司 10001 的分公司列表缓存
10002:user:token:xyz789       -- 公司 10002 的用户 Token
```

#### 3.6.3 文件存储隔离

对象存储路径规范：

```
/{environment}/{tenant_id}/{module}/{yyyy}/{mm}/{uuid}.{ext}
```

示例：
```
/prod/10001/avatar/2026/06/a1b2c3d4.jpg
/prod/10001/document/2026/06/report_e5f6g7h8.pdf
/prod/10002/avatar/2026/06/i9j0k1l2.jpg
```

**文件访问控制**：
- 上传文件时记录 `tenant_id`、`upload_by`、`file_type`
- 访问文件时校验该用户是否有权限访问该文件所属的租户

#### 3.6.4 界面隔离

- **登录页**：公司用户输入账号后，系统识别其所属公司，展示公司 Logo 和名称
- **系统左上角**：始终显示当前公司 Logo + 名称
- **水印**：页面底层显示公司名称水印，防止截图泄露（可选，后续版本）

#### 3.6.5 操作隔离

- 公司 A 的管理员在"人员管理"中搜索，结果仅限公司 A 的人员
- 公司 A 的角色列表中仅显示公司 A 创建的角色
- API 层面：跨租户请求返回 403 Forbidden

#### 3.6.6 业务隔离（后续接入模块时）

- 业务模块（如 OA、财务）开发时必须遵循同样的隔离规范
- 业务数据表纳入 `tenant_id` 体系
- 公司管理员仅能操作本公司的业务数据

#### 3.6.7 超级管理员的跨租户访问

- 超级管理员登录超级后台，不受 `tenant_id` 限制
- 超级管理员查看某公司详情时，后端通过参数传递目标 `tenant_id` 查询数据
- 超级管理员的所有操作均记录在超级管理后台的日志中

### 3.7 登录与认证

#### 3.7.1 登录流程

```
用户输入账号密码
    ↓
系统验证账号密码
    ↓
检查用户状态（启用/停用）
    ↓
检查公司状态（启用/停用/过期）
    ↓
生成 JWT Token
    { tenant_id, user_id, role_codes, exp }
    ↓
返回 Token + 用户信息 + 前端路由权限
```

#### 3.7.2 Token 管理

| 项目 | 规则 |
|------|------|
| Token 有效期 | 2 小时 |
| 刷新机制 | Token 过期前 30 分钟内可刷新，续期 2 小时 |
| 退出登录 | Token 加入 Redis 黑名单，有效期内不可再使用 |
| 强制下线 | 超级管理员停用公司或公司管理员停用用户时，对应 Token 立即加入黑名单 |

#### 3.7.3 登录安全

- [ ] 连续登录失败 5 次，账号锁定 15 分钟
- [ ] 首次登录强制修改密码
- [ ] 密码强度校验（至少 8 位，含大小写字母和数字）
- [ ] 密码加密存储（bcrypt / Argon2，不可明文或可逆加密）
- [ ] 登录日志记录（包含 IP、设备、时间、成功/失败）

#### 3.7.4 多端登录

- 默认允许多端同时在线
- 公司管理员可在系统设置中配置"仅允许单端登录"

#### 3.7.5 前端路由权限

登录成功后，后端根据用户角色返回该用户可访问的路由列表：

```json
{
  "routes": [
    { "path": "/dashboard", "name": "首页" },
    { "path": "/company", "name": "公司管理", "children": [
        { "path": "/company/branch", "name": "分公司管理" },
        { "path": "/company/dept", "name": "部门管理" }
      ]
    },
    { "path": "/system/role", "name": "角色管理" },
    { "path": "/system/user", "name": "人员管理" }
  ],
  "permissions": [
    "company:view", "company:edit",
    "branch:view", "branch:create",
    "dept:view",
    "role:view", "role:create",
    "user:view", "user:create", "user:edit"
  ]
}
```

前端根据路由列表动态注册可访问的路由，无权限的页面在导航菜单中隐藏。

### 3.9 扩展接口预留

**设计原则**：核心系统只做"多租户 + 组织架构 + 权限 + 人员"四件事，其他业务功能（OA、CRM、财务、项目管理等）通过插件/模块方式接入。

#### 3.9.1 模块注册接口

**未来模块接入流程**：
1. 模块开发方按规范实现接口
2. 超级管理员在后台"模块管理"中注册新模块
3. 模块注册后，其权限点自动加入权限树
4. 公司管理员可根据需要给角色分配该模块的权限

**模块注册信息**：

```json
{
  "module_code": "oa_workflow",
  "module_name": "OA 审批流",
  "version": "1.0.0",
  "description": "企业 OA 审批流程模块",
  "entry_url": "/modules/oa-workflow/",
  "permissions": [
    { "code": "oa:template:view", "name": "查看审批模板" },
    { "code": "oa:template:create", "name": "创建审批模板" },
    { "code": "oa:flow:start", "name": "发起审批" },
    { "code": "oa:flow:approve", "name": "审批操作" },
    { "code": "oa:flow:view", "name": "查看审批记录" }
  ],
  "menu_config": {
    "path": "/oa-workflow",
    "name": "OA 审批",
    "icon": "document-checked",
    "children": [
      { "path": "/oa-workflow/start", "name": "发起审批" },
      { "path": "/oa-workflow/todo", "name": "我的待办" }
    ]
  },
  "data_isolation": true,
  "depends_on": ["core:user", "core:dept"]
}
```

#### 3.9.2 统一认证集成

**SSO 单点登录接口（预留）**：

| 接口 | 说明 |
|------|------|
| `/api/sso/authorize` | OAuth2.0 授权端点 |
| `/api/sso/token` | 获取 Token |

## 5. API 接口规范概要

### 5.1 通用规范

| 项目 | 规范 |
|------|------|
| 协议 | HTTPS |
| 格式 | JSON |
| 统一前缀 | `/api/v1/` |

## 6. 安全与合规要求

### 6.1 数据安全

| 要求 | 实现方式 |
|------|----------|
| 传输加密 | 全站 HTTPS，强制 HSTS |
| 密码存储 | bcrypt/Argon2 单向加密，不可逆 |
| SQL 注入防护 | 使用 ORM 参数化查询，禁止拼接 SQL |
| XSS 防护 | 前端输出转义 + CSP 头配置 |
| CSRF 防护 | Token 机制或 SameSite Cookie |
| 文件上传安全 | 白名单格式校验、文件大小限制、病毒扫描（后续对接） |

### 6.2 隐私合规

| 要求 | 说明 |
|------|------|
| 个人信息脱敏 | 前端列表页手机号脱敏显示（如 `138****8888`） |
| 数据最小化 | 仅收集必要字段，用户可查看和导出个人数据 |
| 日志保留策略 | 操作日志超期自动归档或删除（可配置） |
| 数据跨境 | 如涉及跨境部署，需标注数据存储位置 |

### 6.3 访问控制

| 要求 | 说明 |
|------|------|
| 最小权限原则 | 新用户默认赋予最小权限角色 |
| 权限变更即时生效 | 角色权限变更后，用户下次请求时生效（缓存刷新） |
| 接口级权限校验 | 每个 API 接口显式声明所需权限，网关层拦截 |
| IP 白名单（预留） | 公司管理员可配置允许登录的 IP 段 |

## 附录 B：术语表

|------|------|
| 租户（Tenant） | 入驻系统的独立企业/组织 |
