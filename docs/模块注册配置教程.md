# 模块注册（Module Registry）配置教程

> 适用角色：平台超级管理员（平台运营方）
> 入口：平台管理 → 模块管理（侧边栏"业务应用"上方）
> 作用：平台方在这里"上架"一个业务模块，租户登录后就会在侧边栏「业务应用」分组里看到入口，无需改代码。

---

## 一、字段速查表

| 字段 | 中文名 | 类型 | 是否必填 | 当前版本是否真正生效 | 说明 |
|------|--------|------|----------|----------------------|------|
| `moduleCode` | 模块编码 | 字符串 | 是 | ✅ **生效** | 唯一标识；决定渲染哪个内置页 |
| `moduleName` | 模块名称 | 字符串 | 是 | ✅ **生效** | 菜单里显示的标题 |
| `version` | 版本号 | 字符串 | 是 | ⚪ 仅展示 | 形如 `1.0.0`，方便追溯 |
| `entryUrl` | 入口地址 | 字符串 | 是 | ✅ **生效** | 前端路由路径，决定点击跳到哪里 |
| `menuConfig` | 菜单配置 | JSON 字符串 | 是 | ⚠️ **预留，暂未消费** | 预留给多级菜单/图标/排序，填 `[]` 即可 |
| `permissions` | 权限配置 | JSON 字符串 | 是 | ⚠️ **预留，暂未消费** | 预留给按权限点控制可见性，填 `[]` 即可 |

> ⚠️ 注意：`menuConfig` 和 `permissions` 当前只是"规范化的存储字段"——后端会存、前端会校验 JSON 合法性，但**菜单渲染和路由守卫还没读取它们**。所以现在所有已启用的模块，对登录租户都一视同仁地显示。这两个字段是为后续细粒度权限/自定义菜单预留的（详见第六节）。

---

## 二、字段逐项详解

### 1. 模块编码 `moduleCode`

**作用**：模块的唯一 ID。系统靠它判断"这个模块点进去渲染哪个页面"。

**格式规则**：
- 只能由**小写英文字母、数字、下划线**组成，如 `approval`、`attendance`、`crm_v2`
- **全局唯一**：重复会保存失败（后端报"模块编码已存在"）
- 注册后**不能修改**（编辑时该框为禁用状态），想改只能删除重建

**关键规则 —— 内置页映射**：
前端 `ModuleHostView.vue` 用 `moduleCode` 判断渲染内容：

```ts
// src/views/module/ModuleHostView.vue
<ApprovalView v-if="moduleCode === 'approval'" />
<div v-else>该模块内容正在建设中</div>
```

也就是说：
- 填 `approval` → 点进去是真正的 OA 审批中心
- 填其他任何值 → 点进去是"建设中"占位页（等你把对应业务页接入后即可生效）

> 后续每接入一个业务模块，开发只需在 `ModuleHostView.vue` 加一行 `v-else-if="moduleCode === 'xxx'"`，注册时编码写 `xxx` 即可。

---

### 2. 模块名称 `moduleName`

**作用**：侧边栏菜单里显示给用户看的名字。

**格式**：任意可读中文/英文，如 `OA审批中心`、`考勤管理`、`CRM 客户管理`。

**示例**：`OA审批中心`

---

### 3. 版本号 `version`

**作用**：纯展示字段，记录模块版本，方便排障和迭代。

**格式**：语义化版本号 `主.次.修`，如 `1.0.0`。

**示例**：`1.0.0`

---

### 4. 入口地址 `entryUrl`

**作用**：前端路由路径。系统会把该路径动态注册成一个路由，并作为侧边栏菜单项的跳转目标。

**格式规则**：
- 必须以 `/` 开头，如 `/company/approval`
- 建议统一放在 `/company/xxx` 命名空间下，避免和后端已有的硬编码路由（如 `/dashboard`、`/admin/users`）冲突
- 路径中不要带查询参数

**原理**（前端 `router/index.ts`）：
```ts
const relPath = m.entryUrl.replace(/^\//, '')   // "/company/approval" -> "company/approval"
router.addRoute('Main', {
  path: relPath,
  name: 'module-' + m.moduleCode,
  component: () => import('@/views/module/ModuleHostView.vue'),
  meta: { title: m.moduleName, moduleCode: m.moduleCode },
})
```
即：填 `/company/approval` → 自动生成 `/company/approval` 路由 → 指向 `ModuleHostView` → 再由 `moduleCode` 决定渲染 `ApprovalView` 还是占位页。

**示例**：`/company/approval`

---

### 5. 菜单配置 `menuConfig`（预留字段，当前可不填内容）

**作用**：预留给未来"自定义菜单"使用（图标、排序、分组、是否显示等）。

**格式**：合法的 JSON 字符串。当前最低要求是能通过 `JSON.parse` 校验。
- 什么都不配：填 `[]`
- 推荐规范（未来版本会消费）：JSON 数组，每项一个菜单项描述

```json
[
  {
    "title": "审批中心",
    "icon": "Document",
    "sort": 1,
    "group": "业务应用"
  }
]
```

> ⚠️ 当前版本前端**不读取**这个字段，菜单标题只用 `moduleName`、无图标、按注册顺序排序。填 `[]` 完全不影响使用。

---

### 6. 权限配置 `permissions`（预留字段，当前可不填内容）

**作用**：预留给未来"按权限点控制模块可见性/操作权限"使用。

**格式**：合法的 JSON 字符串，数组元素为权限点字符串。
- 什么都不配：填 `[]`
- 推荐规范：模块自带的权限点列表，如

```json
["approval:view", "approval:apply", "approval:approve", "approval:manage"]
```

**当前行为**：所有已登录租户都能在「业务应用」看到已启用的模块，**不会**按 `permissions` 做过滤或拦截。等后续接入前端路由守卫 + 后端 `@PreAuthorize` 细粒度校验后，这里的值才会真正限制"谁能看到/谁能操作"。

> ⚠️ 注意：这跟"模块管理"本身的平台权限是两回事。平台方能否进「模块管理」页面，由后端 `PERM_platform:module:view` 等平台权限点控制（见 `ModuleRegistryController`），与这里填的 `permissions` 无关。

## 三、完整配置示例（以现有「OA审批中心」为例）

这就是系统预置的审批模块在数据库里的真实形态，照抄即可理解：

| 字段 | 值 |
|------|-----|
| 模块编码 | `approval` |
| 模块名称 | `OA审批中心` |
| 版本号 | `1.0.0` |
| 入口地址 | `/company/approval` |
| 菜单配置 | `[]` |
| 权限配置 | `["approval:view","approval:apply","approval:approve","approval:manage"]` |

保存后（默认 `status=1` 启用）→ 租户登录 → 侧边栏「业务应用」出现「OA审批中心」→ 点击进入真实审批页。

## 四、操作流程图

1. 平台超级管理员登录
2. 平台管理 → 模块管理 → 点「注册新模块」
3. 填写六个字段（见上表）
4. 点「确定」→ 后端校验编码唯一 + 各字段非空 + `menuConfig`/`permissions` 为合法 JSON
5. 保存成功，新模块 `status=1`（启用）
6. 租户重新登录（或刷新）→ 侧边栏「业务应用」立即出现新入口
7. 如需下线：列表中「停用」→ `status=0` → 入口从租户菜单消失（动态路由不再注册）

## 五、常见坑 / FAQ

**Q1：为什么我注册的新模块点进去是"建设中"？**
A：只有 `moduleCode = approval` 有内置页。其他编码目前走占位页，需要开发把对应业务页接入 `ModuleHostView.vue`。

**Q2：菜单配置/权限配置填了没反应？**
A：这两个字段当前版本前端未消费，仅做规范化存储。填 `[]` 即可，不影响模块上架和菜单显示。

**Q3：入口地址填错了会怎样？**
A：路由会按你填的路径注册。如果和已有路由重名可能冲突；如果路径非法（不以 `/` 开头）动态路由注册会跳过该项，菜单点了白屏。务必以 `/company/xxx` 形式填写。

**Q4：模块编码写错了能改吗？**
A：不能。编辑时编码框禁用。只能删除重建。

**Q5：普通员工能看到「模块管理」吗？**
A：不能。「模块管理」受平台权限 `PERM_platform:module:*` 保护，只有平台超级管理员可见。租户侧只能看到「业务应用」里已启用的模块入口。

## 六、扩展说明（这两个预留字段将来怎么用）

当你希望"模块注册"从"元数据存储"升级为"真正的动态模块市场"时，开发侧需要补两件事（有需求可直接提）：

1. **菜单配置生效**：前端 `AppSidebar.vue` 读取 `menuConfig`，用其中的 `icon`/`sort`/`group` 渲染带图标、可排序、可归组的菜单项，替代现在直接用 `moduleName` 的简版渲染。
2. **权限配置生效**：
   - 前端路由守卫（`router.beforeEach`）读取 `permissions`，结合用户已有权限点，隐藏无权限的模块入口；
   - 后端在 `ModuleHostView` 对应的业务接口上用 `@PreAuthorize("hasAuthority('PERM_approval:view')")` 等做服务端拦截；
   - 平台方在「角色管理」里把 `approval:view` 之类的权限点分配给租户角色，实现"不同角色看到不同模块"。

届时，`moduleCode` 仍作为唯一标识和内置页映射键，`entryUrl` 仍作为路由路径，而 `menuConfig` 和 `permissions` 就从"占位"变成"真正驱动 UI 与权限"的配置。
